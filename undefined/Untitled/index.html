学习mysql的索引,了解一些硬件的知识会更容易理解一点;

1. 磁盘io读取;

​ 磁盘读取数据靠的是机械运动，每次读取数据花费的时间可以分为寻道时间、旋转延迟、传输时间三个部分;

数据就放在2345中间的扇形区域, 那么要定位到这个扇形区域,就应该知道2345各个位置才行;

​ (1) 寻道: 这里就是定位到4和5的位置;通过指针水平方向移动定位; 磁臂移动到指定磁道所需要的时间，主流磁盘一般在5ms以下；

​ (2) 旋转延迟: 就是这里的2和3,通过旋转就能定位到2和3,速度相对较快, 就是我们经常听说的磁盘转速，比如一个磁盘7200转，表示每分钟能转7200次，也就是说1秒钟能转120次，旋转延迟就是1/120/2 = 4.17ms

​ (3) 传输时间指的是从磁盘读出或将数据写入磁盘的时间，一般在零点几毫秒，相对于前两个时间可以忽略不计。

那么访问一次磁盘的时间，即一次磁盘IO的时间约等于5+4.17 = 9ms左右，听起来还挺不错的，但要知道一台500 -MIPS的机器每秒可以执行5亿条指令;考虑到磁盘IO是非常高昂的操作，计算机操作系统做了一些优化，当一次IO时，不光把当前磁盘地址的数据，而是把相邻的数据也都读取到内存缓冲区内，因为局部预读性原理告诉我们，当计算机访问一个地址的数据的时候，与其相邻的数据也会很快被访问到。每一次IO读取的数据我们称之为一页(page)。具体一页有多大数据跟操作系统有关，一般为4k或8k，

2. 选择什么样的数据结构更适合mysql的存储;

研究数据结构之前可以熟悉下这个网站https://www.cs.usfca.edu/~galles/visualization/Algorithms.html可以很形象的描述各个数据结构;

2.1 哈希

​ hash是我们最容易想到的数据结构,通过hash算法将索引key换算成一个指针地址,也就是行数据存放的位置,这样其实一次io就能解决问题,它的缺陷也显而易见, 当我们执行 a > 100这种范围的查询的时候就一脸懵了,需要一个个遍历了;

​

2.2 二叉树

二叉树的特点: (更我们猜数字的游戏很想,根据大小范围一步步找到)

  1. 每个节点最多有两个子节点,
  2. 每个节点存储了指向数据的指针;
  3. 左子树的值要小于父节点的值,右子树的值要大于父节点的值;

总结: 二叉树的优点很明显极大的减少了查询次数,但是如果数据是顺序增长的,那么树的高度就会很深,查询效率一样很低;

2.3 红黑树

特点: 会自动调整树的结构,保证左右分支的平衡;(具体有相应的算法)

总结: 数据量比较大,比如500w,深度将会很高,还是增加了磁盘IO;

既然要降低树的深度,是不是可以横向扩展,增加每个节点的元素个数;
一次io只读取一个节点的效率是不是太低了,一个节点如果是int的话,只有4b,能不能一次性读取更多个,计算机进行一次磁盘IO,是有最小单位的, 一般是4k,叫做一页, 根据硬件配置的不同读取对应n页, 如果索引是int类型的话, 那么一次最少可以取: 4*1024/4 个元素了;

回想最开始学习的硬件知识,计算机取数据的时候,最小单位是1页,差不多4k的大小,具体一次取多少也跟操作系统有关,也就是最小n*4k的大小,那么用红黑树,一次只取一个值岂不是很浪费, 换一种角度来看, 既然深度不能太深,那么我增加水平的宽带不就可以嘛,每个节点多存点数据;这样b-tree这种数据结构就很合适了;

2.4 B-tree

特点:

  1. 一个节点存多条数据,每条数据对应一个key-value,key是索引值,value是对应的行数据;
  2. 节点中的数据从左到右递增;
  3. 叶子节点没有指针指向;
  4. 叶子节点有相同的深度;

总结: 由于每次查询出一个节点,都是以页为单位,一个节点里面有很多数据,这样极大的减少了树的深度;缺点不太明显哈哈;

  1. 由于行数据也是保存在非叶子节点上,这样同样一页的大小就减少了放索引值的数量,无形中增加了树的深度;

  2. 叶子节点没有指针指向下个节点,做范围查询的时候每次都需要查;

2.5 B+tree

特点:

  1. 通过增加冗余索引,将数据全部放在叶子节点,非叶子节点不存数据;
  2. 非叶子节点直接是有指针指向的,并按照顺序排列;

从它的特点也可以看出,b+tree完美的解决了上面b-tree的两个问题;

  1. 由于非叶子节点不存储数据,一次磁盘的IO取的大小是固定的, 也就是每个节点的大小是固定的, 这就增加了可以存储索引的个数,也就是度; 这样算一下; 如果是树高为4的情况,(4*1024/4)的三次方,也就是十几亿,而且一般根节点通常是放在内存中的,这样一样2次磁盘io就可以从十几亿中查到想要的值, 有些情况二级节点也会放到内存中;这样磁盘io的次数就更少了;
  2. 由于数据全部在叶子节点,那么我们找到一个索引后,就不需要重新查找了,非叶子节点中是有指针的,先从取出来的一页中往后面读,比如我要查大于31的, 图中先查到20,因为20和30在一页中,在查20的时候同时把31也查到内存中了,直接在内存取就行,通过指针,快速找到下一页,40和50,发现40 就不再查询;

这样还不完美

疑问1: 如果有多个索引,一个索引对应一个b+tree,而一个b+tree的叶子节点保存了所有数据data, 这样data数据是不是冗余?

疑问2: 如果data数据有修改,是不是还要全部去修改,保证数据一致性,这样的代价是相当大的;

可以看到图中非主键索引叶子节点对应的不是原始数据而是行数据的主键,这样就完美的解决了上面的两个问题,来回味一下,一是数据没有冗余了,二是也能快速找到数据;,通过主键再去主键树再查询一次,叫做回表;

那么我么写sql语句的时候其实可以做优化了, 当我们只需要查询主键的情况下,用 select id from tt where k; 只查主键的话会减少回表,提升性能;

3. 扩展;

3.1 innodb引擎,mysql一般不会推荐使用业务键作为主键,会让我们设置主键,并且最好是整型,自增长的?

​ (1) 一定要设置主键?
​ 没有主键,非主键索引将无法指向主键了,没有设置的话,mysql也会自动给我们设置;

​ (2) 最好是int类型?

​ 当有多个索引时,多个非主键索引的叶子节点保存的都是主键,如果主键是varchar等其他类型,将会极大的占用了空间; int类型在回表的时候,int类型做比较也是更加性能高的;其实还有一个优势, 如果是字符串其实是需要转成ACII码做比较的,很明显没有数字直接比较快;

​ (3) 为什么要设置主键自增长?
​ 比如varchar等其他类型,通过asci码换算成数字,但是varchar等字符类型时随机的,大小顺序是不能保证的,当插入的大小是中间时,将从中间插入,叶子节点的页中其他元素就要移动位置,如果当前页已经满了,还要做页的分裂,比较损耗性能的,如果是自增长的话,直接在最后面的页中插入数据就可以了;
