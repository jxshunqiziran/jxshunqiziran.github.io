MySQL索引

1. 用哪种数据结构更适合索引?

1. Hash(哈希):

将索引值也就是key通过hash函数换算成一个数值,将value放在数组的这个数值位置,value放的是数据指针;

优点: 快速定位到数据指针,找到数据;

缺点:

  1. 范围查询的话就需要全表扫描,效率较低,比较适合等值查询的场景; HashMap,Redis;

  2. 而且key通过hash函数得到的数值有重复索引的可能(因为数组不可能是无线长的),可能就得加上链表了,一旦数量量比较大,链表就会越长,遍历链表就需要更多的时间;

2. 二叉树:

特点:

  1. 每个节点最多有两个子节点,
  2. 每个节点存储了指向数据的指针;
  3. 左子树的值要小于父节点的值,右子树的值要大于父节点的值;

优点: 极大的减少了查询次数

弊端: 如果插入的数据是顺序增长的,将会出现极端情况; 树的深度很高;

3. 红黑树:

会自动调整树的结构,保证左右分支的平衡;

弊端: 当数据量比较大,比如500w,深度将会很高,还是增加了磁盘IO;

4.B-tree:

  1. 各个节点也存储了数据data的;

  2. 非叶子节点是直接是没有指针的;

    优点: 极大的减少了树的深度;减少了查询次数;

    弊端:

  3. 磁盘IO每次读取到内存的大小是固定页的,innodb通常是一页16k(如果一次磁盘io无线,那岂不是把所有索引放在一个节点做一次磁盘IO岂不是更好),由于除了存索引值,还保存了数据data,这样一个节点就极大减少了能存储索引的个数,也就增加了树的深度

  4. 它的索引值是分散的,做范围查询效率不高,每次都要重新查找;

5. b+tree

特点:

  1. 通过增加冗余索引,将数据全部放在叶子节点,非叶子节点不存数据;

  2. 非叶子节点直接是有指针指向的,并按照顺序排列;

优点分析:

  1. 由于非叶子节点不存储数据,一次磁盘的IO取的大小是固定的, 也就是每个节点的大小是固定的, 这就增加了可以存储索引的个数,也就是度; 这样算一下; 如果是树高为4的情况,(4*1024/4)的三次方,也就是十几亿,而且一般根节点通常是放在内存中的,这样一样2次磁盘io就可以从十几亿中查到想要的值, 有些情况二级节点也会放到内存中;这样磁盘io的次数就更少了;

  2. 由于数据全部在叶子节点,那么我们找到一个索引后,就不需要重新查找了,非叶子节点中是有指针的,先从取出来的一页中往后面读,比如我要查大于31的, 图中先查到20,因为20和30在一页中,在查20的时候同时把31也查到内存中了,直接在内存取就行,通过指针,快速找到下一页,40和50,发现40 就不再查询;

完美解决上面所有问题!!!

人生需要不断的思考:

如果有多个索引,一个索引对应一个b+tree,而一个b+tree的叶子节点保存了所有数据data, 这样data数据是不是冗余?

如果data数据有修改,是不是还要全部去修改,保证数据一致性,这样的代价是相当大的;

这样设计非主键索引的非叶子节点保存的是主键,通过主键再去主键树再查询一次,叫做回表;

===> 所以 当我们只需要查询主键的情况下,用 select id from tt where k; 只查主键的话会减少回表,提升性能;

innodb引擎,mysql一般不会推荐使用业务键作为主键,会让我们设置主键,并且最好是整型,自增长的?

  1. 一定要设置主键?

    没有主键,非主键索引将无法指向主键了,没有设置的话,mysql也会自动给我们设置;

  2. 最好是int类型?

​ 当有多个索引时,多个非主键索引的叶子节点保存的都是主键,如果主键是varchar等其他类型,将会极大的占用了空间;

​ int类型在回表的时候,int类型做比较也是更加性能高的;

  1. 为什么要设置主键自增长?

    比如varchar等其他类型,通过asci码换算成数字,但是varchar等字符类型时随机的,大小顺序是不能保证的,当插入的大小是中间时,将从中间插入,叶子节点的页中其他元素就要移动位置,如果当前页已经满了,还要做页的分裂,比较损耗性能的,如果是自增长的话,直接在最后面的页中插入数据就可以了;
